## Context

xclabel 当前的数据处理流程是离散的：用户需要分别在标注页、训练页、测试页之间手动跳转，无法将「导入 → 标注 → 训练 → 推理 → 导出」串联为一次性执行的自动化链路。Roboflow Workflow 提供了可视化节点编排的方式，让用户可以拖拽节点、配置参数、一键运行。本项目参考此模式，采用轻量化的 JSON 配置 + 表单编辑器方案，避免引入复杂的前端图形库。

## Goals / Non-Goals

**Goals:**
- 每个工程可拥有多个独立工作流配置
- 支持通过表单方式添加/配置/连接工作流节点
- 工作流节点覆盖数据导入、AI 标注、人工审核、训练、推理、导出六大环节
- 支持保存工作流 JSON、运行工作流并查看各节点状态
- 工程首页以标签页形式整合工作流列表入口

**Non-Goals:**
- 不支持可视化拖拽连线（范围过大，先用表单+顺序列表实现）
- 不支持工作流定时触发或 Webhook 触发（后续迭代）
- 不支持跨工程共享工作流模板（后续迭代）
- 不修改现有标注/训练/推理的核心逻辑，仅在工作流层调用已有 API

## Decisions

**Decision 1: 工作流配置格式采用 JSON 而非数据库存储**
- **Rationale**: 与现有工程的 annotations.json、classes.json 等文件式存储保持一致，无需引入数据库依赖
- **Alternative**: SQLite 存储。Rejected：增加部署复杂度，与现有架构不一致

**Decision 2: 前端编辑器采用表单列表而非可视化画布**
- **Rationale**: 降低前端复杂度，当前技术栈（纯 HTML+JS）不便于实现拖拽连线。用「节点列表 + 上移/下移/删除」即可表达顺序执行关系
- **Alternative**: 引入 React Flow 等库。Rejected：引入构建工具和大量依赖，超出当前迭代范围

**Decision 3: 工作流执行采用同步顺序执行（单线程）**
- **Rationale**: 简化状态管理和错误处理。每个节点完成后才执行下一个，便于在 UI 中展示逐节点进度
- **Alternative**: 异步并行执行。Rejected：节点之间常有数据依赖（如训练节点依赖标注节点输出），并行会增加复杂度

**Decision 4: 复用现有 API 而非为工作流重写业务逻辑**
- **Rationale**: 训练、推理、标注等核心逻辑已在 app.py 中实现。工作流执行器通过内部函数调用来复用，保持代码单一职责
- **Trade-off**: 工作流节点配置参数需要与现有 API 参数对齐

## Risks / Trade-offs

- [Risk] 工作流长时间运行导致 HTTP 请求超时 → Mitigation: 工作流运行采用异步执行（类似训练任务的 subprocess/SocketIO 模式），前端通过轮询或 WebSocket 获取状态
- [Risk] 节点执行失败后难以定位和恢复 → Mitigation: 每个节点记录详细日志和执行结果，支持从失败节点重试或回滚到上一个成功节点
- [Risk] 工作流 JSON 结构随功能迭代频繁变更 → Mitigation: 在 JSON 中保留 `version` 字段，后续升级时做向后兼容的迁移
